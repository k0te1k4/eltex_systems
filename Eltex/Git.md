Git - система контроля версий. 
Изменения хранятся в виде копий того же файла, даже если была изменена всего одна строка.

**Команды git не могут использоваться вне папки - репозитория git.**
# Команды
- `git` - показывает основные команды
## Конфигурация
- `git config --global --list` - вывод конфига (почты,имени пользователя)
- `git config --global user.(name|email)` - изменить имя или почту
## Управление репозиторием
- `git init` - создать репозиторий в данной папке
- `git log` - посмотреть историю коммитов
- `git status` - посмотреть текущий статус, изменения в файлах
### Работа с файлами
- `git add file.c` - добавить файл (file.c) в следующий коммит
- `git rm --cached file.c` - убрать файл из текущего (не зафиксированного) коммита
### Работа с коммитами
- `git commit` - создать и зафиксировать коммит
- `git checkout <commit hash>` - вернуть репозиторий к определенному коммиту

# Ветки
Все репозитории при создании имеют основную ветку master/main.

- `git checkout -b <new_branch>` - создать и/или переключиться на ветку new_branch
- `git switch <branch>` - переключиться на другую ветку

## Слияние веток
При слиянии все коммиты из ветки А переходят в ветку Б. 
	Находясь в ветке Б:
	`git merge <branch A>`

Но если в ветке А было 10 коммитов, то в ветке Б окажется только 1 коммит, в котором будут все изменения.
Чтобы история коммитов сохранилась, нужно использовать `git rebase`. Но его нужно использовать с осторожностью, чтобы не возникало дубликатов коммитов.

# Удаленные репозитории
- `git pull` - вытянуть все новые коммиты с удаленного репозитория
- `git push <branch>` - отправить изменения на удаленный репозиторий

## Конфликты
Когда в один и тот же файл вносятся изменения от разных участников в один момент времени.
Для "второго" пользователя, который отправил позже, пишется ошибка с несовпадениями.
После этого один из пользователей должен будет откатить изменения и  принять изменения другого с помощью `git pull`.
